# SongList Studio 产品需求文档

## 1. 产品定位

SongList Studio 是一个面向音乐主播、驻唱、KTV 爱好者的个人歌曲库与歌单生成应用。

用户维护自己会唱的歌曲，系统根据场景和分类维度自动生成清晰、美观、可导出的歌单。产品气质接近手机备忘录或便签软件：轻、快、低学习成本。

一句话定位：

> 管理自己会唱的歌，并快速生成可发布的点歌单。

## 2. 核心判断

这个产品的重点不是播放音乐，也不是复杂设计器，而是解决三个问题：

1. 用户会唱的歌越来越多，存在备忘录里很乱。
2. 每次直播、聚会、演出前重新整理歌单很浪费时间。
3. 用户需要一张能直接发出去的歌单图，但不想自己排版。

所以产品核心链路必须是：

> 录入歌曲 -> 管理曲库 -> 选择生成目标 -> 选择分类维度 -> 编辑歌单 -> 设置背景 -> 导出。

## 3. 目标用户

### 3.1 核心用户

音乐直播主播。

典型场景：

- 直播前快速生成当天点歌单。
- 直播中观众问“你会唱什么”时快速查找。
- 新学一首歌后立刻加入曲库。
- 不同主题直播需要不同歌单，比如“周杰伦专场”“粤语专场”“深夜情歌场”。

### 3.2 次级用户

驻唱、乐队主唱、婚礼歌手、KTV 高频用户。

典型场景：

- 给活动方提供曲目单。
- 给朋友展示自己会唱的歌。
- 按风格或语言准备演唱曲目。

## 4. MVP 范围

MVP 只验证一条完整主流程：

> 用户录入歌曲，选择分类方式，生成歌单，设置背景，导出图片。

### 4.1 MVP 必做

- 歌曲增删改查。
- 批量粘贴导入。
- 曲库搜索和筛选。
- 歌单生成。
- 基础分类维度：歌手、风格、语言、情绪、歌名字数、歌曲状态。
- 歌单编辑：标题、分组名、歌曲排序、隐藏分组、点歌规则。
- 发布历史：保存已发布歌单，支持查看、复制、重新编辑、再次导出。
- 学歌备忘录：记录直播间观众要求主播学歌的事项，支持时间、点歌人、歌名、备注和状态管理。
- 背景设置：纯色、渐变、上传图片。
- 4 套基础皮肤：米白备忘录、粉色便签、深夜模式、牛皮纸。
- PNG 导出。
- 本地数据持久化。
- PC 和手机响应式。

### 4.2 MVP 不做

- 音乐播放。
- 歌词展示。
- 复杂 Canvas 设计器。
- 皮肤市场。
- OBS 插件。
- 直播间实时点歌队列。
- 社区广场。
- 云同步。
- 账号系统。
- 大规模版权曲库。

这些功能会把产品带偏，且不影响验证核心价值。

说明：

- MVP 可以做“学歌备忘录”，但不做完整“直播间实时点歌队列”。
- 学歌备忘录只解决承诺记录、待学跟进和学会后入库；不处理礼物识别、观众排队、实时弹幕同步和直播平台对接。

## 5. 页面结构

### 5.1 首页

目标：让用户快速进入常用动作。

内容：

- 曲库歌曲总数。
- 最近生成的歌单。
- 最近发布的歌单。
- 快速按钮：添加歌曲、新建歌单、批量导入。
- 推荐生成入口：直播点歌单、KTV 歌单、练习歌单、主题歌单。

关键要求：

- 首屏不要做营销介绍。
- 首页要像工具台，不像落地页。

### 5.2 我的曲库

目标：管理用户会唱的所有歌曲。

功能：

- 歌曲列表。
- 搜索歌名、歌手。
- 按歌手、风格、语言、情绪、状态筛选。
- 添加歌曲。
- 编辑歌曲。
- 删除歌曲。
- 批量选择。
- 批量修改标签或状态。

歌曲卡片展示：

- 歌名。
- 歌手。
- 风格标签。
- 语言标签。
- 状态标签。

### 5.3 添加歌曲

目标：降低录入成本。

支持两种方式：

1. 单首添加。
2. 批量粘贴导入。

单首字段：

- 歌名，必填。
- 歌手，选填但建议填写。
- 风格。
- 语言。
- 情绪。
- 状态。
- 熟练度。
- 备注。

批量导入格式：

```text
稻香 - 周杰伦
后来 - 刘若英
十年 - 陈奕迅
小幸运
```

解析规则：

- 识别 `歌名 - 歌手`。
- 识别 `歌名 / 歌手`。
- 无法识别歌手时，只导入歌名。
- 导入后进入确认页，用户可以批量修正。

### 5.4 新建歌单

目标：从曲库生成一个可编辑歌单。

流程：

1. 选择生成目标。
2. 选择歌曲范围。
3. 选择分类维度。
4. 预览分组结果。
5. 进入歌单编辑器。

生成目标：

- 直播点歌单。
- KTV 歌单。
- 演出曲目单。
- 练习歌单。
- 主题歌单。

歌曲范围：

- 全部歌曲。
- 仅可点歌。
- 指定风格。
- 指定语言。
- 指定标签。
- 排除练习中或暂不唱。

### 5.5 歌单编辑器

目标：让用户把生成结果调整到可发布状态。

功能：

- 编辑歌单标题。
- 编辑副标题，比如日期、主播名、场次主题。
- 编辑点歌规则。
- 修改分组名称。
- 隐藏或显示分组。
- 调整分组顺序。
- 调整组内歌曲顺序。
- 删除某首歌。
- 添加临时歌曲。
- 设置背景。
- 导出图片。
- 发布为历史歌单。

关键原则：

- 编辑器不是设计器。
- 用户只能做有限但高频的调整。
- 所有样式选择都应该是一键应用。

### 5.6 发布历史 / 歌单库

目标：管理用户之前发布过的歌单，方便查看、复用和再次发布。

这个模块必须进入 MVP。原因是用户发布过的歌单不是临时结果，而是作品记录。没有历史管理，用户每次都要重新生成，产品会退化成一次性工具。

功能：

- 查看已发布歌单列表。
- 查看歌单详情。
- 按标题、日期、场景搜索。
- 按生成目标筛选：直播点歌单、KTV 歌单、演出曲目单、练习歌单、主题歌单。
- 复制一个历史歌单。
- 基于历史歌单重新编辑。
- 用当前曲库重新生成。
- 再次导出图片。
- 删除历史记录。
- 标记常用歌单。
- 将历史歌单保存为模板。

列表展示：

- 歌单封面缩略图。
- 歌单标题。
- 发布日期。
- 生成目标。
- 分类维度。
- 歌曲数量。
- 最近一次导出时间。

状态区分：

- 草稿：生成后还未发布或导出。
- 已发布：用户点击“发布/保存到历史”或成功导出后进入发布历史。
- 模板：用户主动标记为可复用配置。

关键交互：

- 从歌单编辑器点击“发布”后，生成一条历史记录。
- 成功导出 PNG 后，如果当前歌单未发布，系统提示保存到发布历史。
- 历史歌单打开后默认只读，点击“编辑副本”才进入编辑器。
- 重新编辑历史歌单时，不覆盖原记录，默认生成新版本。

### 5.7 学歌备忘录

目标：记录直播间观众要求主播学歌的事项，避免主播忘记承诺，也方便后续把已学会歌曲加入曲库。

这个模块应该进入主播版 MVP。原因是主播场景里“观众刷礼物要求学歌”是高频真实事件，但它不等同于直播间实时点歌队列，产品边界更轻。

功能：

- 快速新增一条学歌记录。
- 记录时间，默认使用当前时间，允许手动修改。
- 记录点歌人或观众昵称。
- 记录歌名，必填。
- 记录歌手，选填。
- 记录礼物或来源说明，选填。
- 记录备注，例如版本、片段、承诺时间。
- 设置状态：待学、练习中、已学会、已放弃。
- 按状态筛选。
- 按点歌人、歌名搜索。
- 将“已学会”的歌曲一键加入曲库。
- 加入曲库后保留原始学歌记录，方便追溯来源。

列表展示：

- 歌名。
- 歌手。
- 点歌人。
- 记录时间。
- 状态。
- 备注摘要。

关键交互：

- 首页提供“记一首待学歌”的快速入口。
- 新增记录时只要求填写歌名，其他字段可以后补。
- 当状态改为“已学会”时，系统提示是否加入曲库。
- 如果曲库中已有同名歌曲，系统提示合并或只更新状态，避免重复入库。

不做：

- 不识别直播间礼物。
- 不接入弹幕。
- 不做观众排队。
- 不做支付或打赏统计。
- 不自动承诺学习截止时间。

### 5.8 设置

目标：承载低频配置。

功能：

- 默认背景。
- 默认皮肤。
- 标签管理。
- 数据导入。
- 数据导出。
- 清空本地数据。

## 6. 核心数据结构

### 6.1 Song

```ts
type Song = {
  id: string
  title: string
  artist?: string
  genre?: string
  language?: string
  mood?: string
  status: 'requestable' | 'practicing' | 'paused'
  proficiency?: 'rough' | 'complete' | 'familiar' | 'signature'
  tags: string[]
  note?: string
  createdAt: number
  updatedAt: number
}
```

状态说明：

- `requestable`：可点歌。
- `practicing`：练习中。
- `paused`：暂不唱。

熟练度说明：

- `rough`：会一点。
- `complete`：能完整唱。
- `familiar`：熟练。
- `signature`：拿手歌。

### 6.2 Playlist

```ts
type Playlist = {
  id: string
  title: string
  subtitle?: string
  purpose: 'live' | 'ktv' | 'performance' | 'practice' | 'theme'
  categoryDimension: CategoryDimension
  sections: PlaylistSection[]
  rulesText?: string
  skinId: string
  background: BackgroundConfig
  lifecycleStatus: 'draft' | 'published' | 'template'
  thumbnail?: string
  publishedAt?: number
  lastExportedAt?: number
  sourcePlaylistId?: string
  createdAt: number
  updatedAt: number
}
```

字段说明：

- `lifecycleStatus`：区分草稿、已发布、模板。
- `thumbnail`：歌单缩略图，用于历史列表展示。MVP 可存储导出图片的压缩 Data URL。
- `publishedAt`：用户保存到发布历史的时间。
- `lastExportedAt`：最近一次导出图片的时间。
- `sourcePlaylistId`：如果从历史歌单复制而来，记录来源歌单。

### 6.3 PlaylistSection

```ts
type PlaylistSection = {
  id: string
  title: string
  songIds: string[]
  hidden: boolean
  order: number
}
```

### 6.4 BackgroundConfig

```ts
type BackgroundConfig = {
  type: 'solid' | 'gradient' | 'image'
  value: string
  opacity?: number
  blur?: number
}
```

### 6.5 CategoryDimension

```ts
type CategoryDimension =
  | 'artist'
  | 'genre'
  | 'language'
  | 'mood'
  | 'titleLength'
  | 'status'
```

### 6.6 LearningRequest

```ts
type LearningRequest = {
  id: string
  songTitle: string
  artist?: string
  requesterName?: string
  requestedAt: number
  sourceNote?: string
  giftNote?: string
  note?: string
  status: 'todo' | 'practicing' | 'learned' | 'abandoned'
  linkedSongId?: string
  createdAt: number
  updatedAt: number
}
```

字段说明：

- `songTitle`：观众要求主播学习的歌名，必填。
- `requesterName`：点歌人或观众昵称。
- `requestedAt`：直播间记录时间，默认当前时间。
- `sourceNote`：来源说明，例如直播场次或平台。
- `giftNote`：礼物或触发原因的文字备注，不做金额统计。
- `status`：待学、练习中、已学会、已放弃。
- `linkedSongId`：当歌曲加入曲库后，记录对应曲库歌曲 ID。

## 7. 分类规则

### 7.1 按歌手

规则：

- 同一歌手归为一组。
- 未填写歌手归入“其他”。
- 只有 1 首歌的歌手可合并到“其他歌手”。

适合：

- 歌手专场。
- 观众熟悉歌手名的场景。

### 7.2 按风格

规则：

- 按 `genre` 字段分组。
- 未填写风格归入“未分类”。

推荐基础风格：

- 流行。
- 民谣。
- 摇滚。
- R&B。
- 古风。
- 粤语。
- 影视 OST。
- 说唱。
- 其他。

适合：

- 直播点歌单。
- 演出曲目单。

### 7.3 按语言

规则：

- 按 `language` 字段分组。

推荐语言：

- 国语。
- 粤语。
- 英语。
- 日语。
- 韩语。
- 其他。

适合：

- 多语种主播。
- 主题歌单。

### 7.4 按情绪

规则：

- 按 `mood` 字段分组。

推荐情绪：

- 甜。
- 伤感。
- 安静。
- 热场。
- 治愈。
- 炸场。

适合：

- 深夜场。
- 主题直播。
- KTV 氛围歌单。

### 7.5 按歌名字数

规则：

- 1-2 字：短歌名。
- 3-4 字：常规歌名。
- 5-7 字：长歌名。
- 8 字以上：超长歌名。

适合：

- 趣味分类。
- 直播互动。

### 7.6 按状态

规则：

- 可点歌。
- 练习中。
- 暂不唱。

适合：

- 主播内部管理。
- 练习歌单。

## 8. 生成目标与推荐分类

不同目标下，系统推荐不同分类维度。

### 8.1 直播点歌单

推荐优先级：

1. 按风格。
2. 按情绪。
3. 按歌手。
4. 按语言。

原因：

直播观众通常不是按数据库逻辑找歌，而是按“我现在想听什么感觉”来点歌。

### 8.2 KTV 歌单

推荐优先级：

1. 按情绪。
2. 按歌手。
3. 按语言。
4. 按熟练度。

原因：

KTV 更强调气氛和大家会不会跟唱。

### 8.3 演出曲目单

推荐优先级：

1. 按风格。
2. 按语言。
3. 按歌手。

原因：

演出曲目单需要显得专业、清楚，不需要太多趣味分类。

### 8.4 练习歌单

推荐优先级：

1. 按状态。
2. 按熟练度。
3. 按风格。

原因：

练习场景关注的是哪些歌还不稳。

### 8.5 主题歌单

推荐优先级：

1. 按情绪。
2. 按风格。
3. 按语言。
4. 按歌名字数。

原因：

主题歌单可以更有趣，适合直播活动和内容包装。

## 9. 背景与皮肤

### 9.1 背景设置

MVP 支持：

- 纯色。
- 渐变。
- 上传图片。

图片背景要求：

- 支持透明度调节。
- 支持轻微模糊。
- 歌单文字必须始终可读。

### 9.2 MVP 皮肤

第一版只做 4 套：

1. 米白备忘录。
2. 粉色便签。
3. 深夜模式。
4. 牛皮纸。

每套皮肤控制：

- 背景默认值。
- 字体颜色。
- 分组标题样式。
- 卡片透明度。
- 边框颜色。
- 强调色。

### 9.3 设计原则

- 不做复杂排版工具。
- 不允许用户自由拖动文本框。
- 不提供过多字体选择。
- 背景是个性化，不是主功能。

## 10. 导出规格

MVP 导出 PNG。

规格：

- 默认 9:16 竖版。
- 支持 1:1 方图作为后续版本。
- 导出 2 倍高清。
- 图片内包含标题、分组、歌曲、点歌规则。

导出前校验：

- 至少有 1 首歌。
- 至少有 1 个可见分组。
- 歌单标题不能为空。
- 文字不能明显溢出。

## 11. 技术建议

### 11.1 前端

- React + TypeScript + Vite。
- Tailwind CSS 或普通 CSS 变量。
- Zustand 管理状态。
- Dexie.js 管理 IndexedDB。
- dnd-kit 实现拖拽排序。
- html-to-image 实现 PNG 导出。

### 11.2 数据存储

MVP 使用 IndexedDB。

原因：

- 不需要登录。
- 实现快。
- 用户数据不小，localStorage 不适合。
- 后续可以迁移到云端。

### 11.3 渲染方案

使用 HTML + CSS 渲染歌单，然后用 `html-to-image` 导出。

不建议 MVP 使用 Canvas 编辑器。

原因：

- Canvas 会增加复杂度。
- HTML + CSS 更适合便签风格。
- 所见即所得更容易保证。
- 文本排版和响应式更简单。

## 12. 开发优先级

### 第一阶段：完整 MVP

目标：

> 一个真实用户能完成从建库到导出歌单图的完整流程。

功能：

- 基础页面框架。
- 曲库增删改查。
- 批量导入。
- 本地存储。
- 分类生成。
- 歌单编辑器。
- 发布历史 / 歌单库。
- 学歌备忘录。
- 背景设置。
- PNG 导出。

### 第二阶段：提升效率

功能：

- 模板复用。
- 更完整的筛选。
- 更多皮肤。
- 文本导出。
- 数据备份和恢复。

### 第三阶段：智能化

功能：

- AI 主题分组。
- 自动补全歌曲元数据。
- 根据场景推荐分类。
- 相似歌曲替换。

### 第四阶段：云端与分享

功能：

- 账号系统。
- 云同步。
- 分享链接。
- 公开歌单页。

## 13. 首次使用流程

新用户必须在 3 分钟内看到第一张歌单。

流程：

1. 输入昵称或主播名。
2. 批量粘贴至少 5 首歌。
3. 系统自动生成基础分类。
4. 用户选择一套背景。
5. 导出第一张歌单。

关键要求：

- 不要求用户一开始填写完整字段。
- 缺失信息可以后续补。
- 空字段不能阻断生成流程。

## 14. 成功指标

MVP 阶段重点看行为指标：

- 用户是否能成功导入歌曲。
- 用户是否能生成第一张歌单。
- 用户是否会回来继续添加歌曲。
- 用户是否会保存多个歌单。
- 用户是否会查看或复用历史发布歌单。
- 用户是否会导出图片。

建议指标：

- 首次导入完成率。
- 首张歌单生成率。
- PNG 导出率。
- 发布历史保存率。
- 历史歌单复用率。
- 平均曲库歌曲数。
- 7 日内新增歌曲次数。

## 15. 当前最重要的产品决策

1. 先做“个人曲库 + 歌单生成”，不要做播放器。
2. 先做“背景设置 + 预设皮肤”，不要做完整设计器。
3. 先做本地存储，不做账号系统。
4. 先做用户手动录入和批量导入，不依赖大型歌曲数据库。
5. 分类功能要围绕使用场景推荐，而不是只堆维度。
